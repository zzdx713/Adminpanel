## 二、痛点分析

### 2.1 严重程度：高

**P0-1：多用户体系缺失**
- 当前仅支持单用户本地密码登录
- 无 RBAC（角色权限控制），所有用户等价超级管理员
- 无审计日志，操作行为无法追溯
- 无 SSO/OAuth 企业认证集成能力

**P0-2：告警与通知体系缺失**
- 系统异常（如 Gateway 断开、任务失败、Agent崩溃）无实时告警
- 无统一的通知中心，用户无法及时获知重要事件
- Cron 任务执行结果仅能在页面查看，无推送能力

**P0-3：Office / MyWorld 为空壳**
- 这两个模块在 UI 上有入口和布局，但无实际后端功能
- 属于"占位模块"，用户期望与实际功能严重不符

### 2.2 严重程度：中

**P1-1：数据可视化颗粒度粗**
- Dashboard 仅支持 today/7d/30d 三个预设时间范围，无自定义高级筛选
- 趋势图 hover 查看日详情，但无下钻能力（点进去看某天具体发生了什么）
- Top 排行只有数量/Token，无响应时长、错误率等质量指标

**P1-2：批量操作能力弱**
- Session 管理无批量删除/导出
- Skills 无批量启用/禁用
- Channels 无批量启停
- 模型配置需逐个手动输入，无批量导入

**P1-3：文件管理功能原始**
- 不支持远程文件 diff / merge
- 不支持文件版本历史
- 不支持大文件分片上传

**P1-4：定时任务配置体验差**
- Cron 表达式无可视化生成器
- 无执行预览/试运行
- 表达式写错无校验反馈

**P1-5：模型配置风险高**
- API Key 输入后直接落库，无测试验证流程
- 无模型调用沙箱/试跑
- 价格/配额预警缺失

### 2.3 严重程度：低

**P2-1：国际化不完整**
- 仅中英文，部分文本硬编码
- 切换语言后部分组件/弹窗仍为中文

**P2-2：无 API Token 管理**
- 无对外开放 API 能力
- 无法供其他系统集成调用

**P2-3：移动端体验差**
- 纯桌面设计，移动端无法正常使用
- 无 PWA 支持

**P2-4：帮助文档缺失**
- 无交互式新手引导
- 无工具提示（Tooltips）
- 错误信息不够友好

## 四、优先级排序的需求列表

### 🔴 P0 - 立即执行（当前 sprint）

| # | 需求 | 背景与目标 |
|---|------|-----------|
| R-01 | **实现多用户 + RBAC 权限体系** | 企业级必备；当前任何人都能操作所有功能，风险极高。需要支持：管理员/操作员/只读三种角色，敏感操作（删除/重置）需二次确认或仅管理员可执行。 |
| R-02 | **实现通知中心 + 告警渠道** | 系统异常用户无法及时获知；需覆盖：Gateway 断开、任务执行失败、Agent崩溃、Token用量超阈值、计划外支出等场景。告警渠道优先支持 飞书/钉钉/邮件。 |
| R-03 | **完善 Office 智能体工坊基础功能** | 当前为占位空壳，用户点击后无响应，体验断裂。至少实现：多 Agent 场景创建、任务委派流程、Agent 间消息传递的基础链路。 |
| R-04 | **完善 MyWorld 虚拟公司基础功能** | 同上，需有实际可用的核心流程，不能只是静态 UI。 |

### 🟠 P1 - 近期迭代（下一个 sprint）

| # | 需求 | 背景与目标 |
|---|------|-----------|
| R-05 | **Dashboard 时间范围自定义 + 下钻能力** | 当前 7d/30d 预设不够用；用户需要看具体某天的会话详情、错误分布。 |
| R-06 | **Cron 可视化编辑器 + 试运行** | Cron 表达式有学习门槛，可视化能降低配置错误率；试运行让用户在正式执行前验证逻辑。 |
| R-07 | **Session / Skills / Channels 批量操作** | 运维场景中经常需要批量清理或批量启停，当前必须逐个操作，效率低。 |
| R-08 | **模型配置增加测试验证流程** | API Key 配置后应先探测是否可用再保存；增加"测试调用"按钮，给出明确的成功/失败反馈。 |
| R-09 | **Webhook 支持** | 支持将任务执行结果、告警事件等以 Webhook 形式推送到用户指定地址，实现与外部系统的自动化联动。 |
| R-10 | **成本预警 + 用量配额管理** | 用户需要知道本月预计花费、接近免费额度时预警；避免月末账单 surprises。 |

## 五、反共识观点（基于产品视角）

1. **不要在 P0 阶段做 Office/MyWorld 的"完美实现"** —— 应该先做最小可用链路（能跑通一个多 Agent 协作任务即可），避免过度设计。先验证用户是否真的需要这个功能，再投入资源完善。

2. **告警不要贪多** —— 第一期只做最高频的 3~5 种告警（Gateway 断开、任务失败、Token 超阈值），不要试图做一个"告警规则引擎"，那是一个独立的产品。

3. **RBAC 不要过度设计** —— 初期 3 个角色（管理员/操作员/只读）足够；不要设计复杂的自定义角色和权限颗粒度，那只会拖慢上线且增加维护成本。

4. **Webhook 优先于邮件/钉钉告警渠道** —— Webhook 是最通用的集成方式，用户收到 webhook 后可以自己转发到任意平台（Slack/飞书/钉钉等）；直接做飞书/钉钉渠道是定制化开发，性价比低。
